# 自动文本 AI 降率工具

本地 Web 应用，用于教材 AIGC 痕迹消除的人工审批工作流，支持一审（降率）与二审（质量校验）两道流程。

上传 `.docx` → AI 率检测 → 逐段审校（一审）→ 人工批准/编辑/跳过 → **二审质量校验** → 导出 Word。

---

## 功能概览

| 功能 | 说明 |
|------|------|
| 文档解析 | 按编号节标题（`1.` / `1.1` / `1.2` 等）及 Word 标题样式将段落分组，**完整保留原始字体/字号/间距等格式** |
| 中英文识别 | 自动区分中文、英文、标题段落，左侧面板分色显示 |
| **AI 率检测** | 独立 API 密钥调用 DeepSeek，对每段打分（0–100），圆环动画展示；≤ 60% 建议跳过，> 60% 自动触发审校 |
| 流式审校（一审） | 调用 DeepSeek `deepseek-chat` 实时流式输出，打字机效果；聚焦 AIGC 痕迹消除与学术规范 |
| Diff 高亮 | 审校完成后，修改处黄色高亮对比原文 |
| 四种操作（一审） | ✅ 批准 / 🔄 重新生成（temperature 递增）/ ✏️ 手动编辑 / ⏭️ 保留原文 |
| 操作历史 | 右侧面板记录每组决策与 AI 率，可点击回溯查看原文 + 修改对比 |
| **暂停 / 继续** | 随时暂停，进度自动保存到 `localStorage`；关闭浏览器后重新打开可一键恢复 |
| **导出审阅版** | 原文黑色删除线 + 修改版青绿色斜体（`RGB 0,134,134`），逐段对照，完整保留原文档格式 |
| **导出纯净版** | 仅最终内容，无任何标记，格式与原文一致；适合直接交付 |
| **全自动审阅** | 一键开启批处理：AI 率 > 60% 自动审校并批准，≤ 60% 自动跳过；实验性功能，启动前有确认提示，随时可中断 |
| **🔍 二审校验** | 一审完成后进入二审，逐段 AI 深度分析；聚焦语言质量、知识硬伤与逻辑连贯性，给出修改建议+理由，由作者决定是否采用 |

- Python 3.10+

## 快速开始

### 1. 安装依赖

```bash
pip install -r requirements.txt
```

### 2. 启动

```bash
python app.py
```

浏览器打开 [http://localhost:5000](http://localhost:5000)

### 3. 使用流程

#### 一审（AIGC 降率）

1. **填写 API 密钥**（首页或右上角）
   - **审校密钥**：用于 DeepSeek 改写，必填
   - **检测密钥**：用于 AI 率评分，可与审校密钥相同，留空则跳过检测直接审校
   - **二审密钥**：用于二审质量校验，可提前填写或进入二审后再填
2. **上传 `.docx` 文件**（拖拽或点击选择）
3. 系统对每组段落先检测 AI 率（圆环显示），再决定是否自动触发审校
4. 对每组结果做出决策（支持键盘快捷键）：

   | 按键 | 操作 |
   |------|------|
   | `Enter` | 批准采用 AI 修改 |
   | `R` | 重新生成（temperature +0.15，上限 0.9） |
   | `E` | 手动编辑（`Ctrl+Enter` 保存，`Esc` 取消） |
   | `S` | 保留原文跳过 |

5. 可随时点击 **⏸ 暂停** 暂停工作：
   - 进入暂停页面，**📥 导出审阅版** / **📄 导出纯净版** / **▶ 继续审校** / **🔄 重新开始** / **🔍 进入二审**
6. 全部处理完毕后，选择 **📥 导出审阅版** / **📄 导出纯净版** 或 **🔍 进入二审**

#### 二审（语言·知识·逻辑校验）

二审在一审完成后（或暂停时对已处理段落）自动衔接，独立使用同一套 DeepSeek 密钥（可另配专属密钥）。

**二审聚焦三个维度：**
- **语言准确、通顺与流畅**：用词准确性、句子结构、英文语法与地道度
- **知识硬伤与实务逻辑**：专业术语（Incoterms、UCP 600 等）、业务流程描述、中英术语对应
- **逻辑连贯性**：段落内部自洽、句间逻辑关系、论述完整性

**操作方式：**
- AI 逐段给出**修改建议**（完整改写后文本）+ **修改理由**（逐条列明问题位置与原因）
- 若当前段落无明显问题，AI 返回「无需修改」

| 按键 | 操作 |
|------|------|
| `Enter` | 采用建议（有建议时）/ 确认跳过（无需修改时） |
| `E` | 手动编辑（以建议文本为基础） |
| `S` | 忽略建议，保留当前文本 |

二审完成后，选择 **📥 导出审阅版**（`s2_revised.docx`）或 **📄 导出纯净版**（`s2_clean.docx`）。

## 导出格式说明

### 审阅版（`revised.docx` / `s2_revised.docx`）

| 段落情况 | Word 样式 |
|----------|-----------|
| 批准 / 手动编辑 | 原文（黑色删除线）紧接修改版（青绿色斜体，`RGB 0,134,134`） |
| 跳过 / 忽略 / 未处理段落 | 仅输出当前文本，无删除线，完整保留原格式 |
| 组间分隔 | 一个空行（间距与原文一致） |

### 纯净版（`revised_clean.docx` / `s2_clean.docx`）

| 段落情况 | Word 样式 |
|----------|-----------|
| 批准 / 手动编辑 | 仅输出修改后文本，按位置索引映射回原段落样式和格式 |
| 跳过 / 忽略 / 未处理段落 | 原文原样输出 |

**两种格式共同保留**：段落样式名、字体（ASCII + 东亚/CJK 双通道）、字号、粗体/斜体、段前/段后间距、首行缩进、对齐方式。

## AI 率检测说明

检测模型为 `deepseek-chat`（temperature 0.1，返回 JSON），评分起点 10 分，宁可高判不漏判。

**中文极高权重信号（+18–25 分/处）**：过度分点（`①②③`、`（一）（二）`、`·` 并列、短行列举 ≥ 3 条）、链式枚举（首先/其次/再次/最后）、套路收尾（综上所述）、虚置引导语（值得注意的是）、空洞评价叠用

**英文极高权重信号（+18–25 分/处）**：`First/Second/Third` 链条、`It is worth noting that`、`In summary,`、`plays a crucial role in`（≥ 2 处）、`not only…but also` 机械对仗

## 审校系统提示词核心规则

### 一审（AIGC 降率）优先级从高到低：

1. **零号规则（绝对优先）**：条目结构（`①②③`、`·` 并列、短行列举等）必须彻底解构后重新拟写为学术散文，禁止逐条润色，禁止"首先/其次"改写，必须提取逻辑关系后重新融合为连贯段落
2. **AIGC 痕迹消除**：结构性套话、评价空话、模板化句式、英文 AI 固定搭配
3. **学术性与严谨性**：中英文专业术语规范、逻辑论证内在自洽
4. **文体规范**：正式书面语、GB/T 15834 标点、第三人称客观视角

### 二审（质量校验）审校维度：

1. **语言维度**：中文用词准确、句式清晰流畅；英文语法正确、用词地道
2. **知识维度**：专业术语（Incoterms 2020/UCP 600/CISG）准确，业务流程描述合理，中英术语一致
3. **逻辑维度**：段落内部自洽，句间逻辑关系清晰，论述无断层

## 技术栈

| 层 | 技术 |
|----|------|
| 后端 | Python + Flask，SSE 流式推送，无数据库 |
| 前端 | Vue 3（CDN）+ Tailwind CSS（CDN Play），单 HTML 文件 |
| 文档处理 | python-docx，含 `w:eastAsia` XML 节点字体保留 |
| AI | DeepSeek API，`deepseek-chat` |
| 状态持久化 | 浏览器 `localStorage`（进度 + 三个 API 密钥） |

## API 端点一览

| 端点 | 方法 | 请求头 | 说明 |
|------|------|--------|------|
| `/` | GET | — | 前端页面（绕过 Jinja2，直接返回文件） |
| `/health` | GET | — | 健康检查 |
| `/upload` | POST | — | 上传并解析 `.docx`，返回段落组 JSON（含格式数据） |
| `/review` | POST | `X-API-Key` | 一审：DeepSeek 审校，SSE 流式返回 |
| `/detect` | POST | `X-API-Key-Detect` | AI 率检测，返回 `{rate, indicators, confidence}` |
| `/proofread` | POST | `X-API-Key-Proofread` | 二审：语言/知识/逻辑深度校验，SSE 流式返回 |
| `/export` | POST | — | 生成并返回对照 `.docx`（一审和二审共用） |

## 安全说明

- 密钥**不写入任何服务器文件**，仅存于浏览器 `localStorage`
- 每次请求通过 HTTP 请求头传入后端，后端直接转发给 DeepSeek，服务端不留存
- 建议仅在本地或可信局域网使用，**不要暴露到公网**

## 常见问题

**Q: 段落组很少，很多内容没被识别到？**
依赖编号标题（`1.` `1.1`）或 Word 标题样式（Heading 1/2/3）分组。纯正文无标题将整体作为一组。

**Q: AI 率显示 50%，置信度 low？**
段落过短（< 40 字）或仅含标题行时返回中性分值，可忽略检测直接手动判断。

**Q: 重新生成多次效果相似？**
temperature 上限 0.9，建议超过 3 次后切换手动编辑模式直接修改。

**Q: 导出 Word 后字体与原文略有差异？**
工具读取第一个有内容的段落运行（run）的字体。若原段落内有多种字体混排，当前版本取首个运行样式作为代表格式。

**Q: 二审与一审可以用同一个密钥吗？**
可以。三个密钥（审校/检测/二审）均可填写同一个 DeepSeek API 密钥，也可按需分配不同账号的密钥以控制用量。

**Q: 二审有进度保存吗？**
当前版本二审无 `localStorage` 持久化，刷新页面后二审进度不保留。建议在同一会话内完成二审后再导出。
